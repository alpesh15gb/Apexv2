# Apexv2 ‚Äî 17 Feature Implementation Plan

## Phase A: Core HR (5 features)

### A1. Documents & Letters
**Model:** `Document` ‚Äî tenant_id, employee_id (nullable), doc_type (offer_letter/experience_letter/id_proof/address_proof/certificate/other), title, file_path, file_name, file_size, mime_type, uploaded_by, is_confidential, expiry_date
**Endpoints:** CRUD + upload/download
**Frontend:** Document list per employee + company-wide document library

### A2. Employee Onboarding
**Model:** `OnboardingTask` ‚Äî tenant_id, employee_id, title, description, assigned_to, due_date, status (pending/in_progress/completed/skipped), completed_at, order_index
**Endpoints:** Create onboarding checklist template, assign to employee, update tasks
**Frontend:** Onboarding checklist view on employee detail page

### A3. Employee Exit
**Model:** `ExitRequest` ‚Äî tenant_id, employee_id, resignation_date, last_working_date, reason, status (pending/approved/rejected/completed), approved_by, exit_interview_notes, clearance_status
**Endpoints:** CRUD + approve/reject + mark complete
**Frontend:** Exit request list + clearance checklist

### A4. Access Roles UI
**Existing models:** Role, Permission, user_roles, role_permissions already exist
**Missing:** Frontend screens to manage roles, assign permissions, assign roles to users
**Frontend:** Role list screen, Role edit screen with permission checkboxes, User role assignment

### A5. Employee Timeline
**Model:** `EmployeeEvent` ‚Äî tenant_id, employee_id, event_type (joining/promotion/transfer/salary_change/leave/attendance_warning/other), title, description, event_date, created_by
**Endpoints:** List events per employee, create event
**Frontend:** Timeline widget on employee detail page

## Phase B: Payroll & Finance (6 features)

### B1. Expense Management
**Models:**
- `ExpenseCategory` ‚Äî tenant_id, name, code, description, is_active
- `ExpenseClaim` ‚Äî tenant_id, employee_id, category_id, amount, date, description, receipt_path, status (draft/submitted/approved/rejected/reimbursed), approved_by
**Endpoints:** CRUD + approve/reject
**Frontend:** Expense claim form, expense list with filters, approval screen

### B2. Gratuity Management
**Model:** `GratuityConfig` ‚Äî tenant_id, eligibility_years (default 5), formula (15_days_salary_per_year or custom)
**Logic:** Calculate gratuity = (last_salary * 15/26) * years_of_service
**Endpoints:** Calculate gratuity per employee, generate report
**Frontend:** Gratuity calculator on employee detail page

### B3. Tax Management
**Model:** `TaxDeclaration` ‚Äî tenant_id, employee_id, financial_year, hra_received, rent_paid, section_80c, section_80d, home_loan_interest, other_exemptions, status (draft/submitted/verified)
**Endpoints:** CRUD + verify
**Frontend:** Tax declaration form, tax computation view

### B4. Perks & Benefits
**Model:** `Benefit` ‚Äî tenant_id, name, type (allowance/reimbursement/benefit), amount, frequency (monthly/quarterly/yearly), is_taxable, is_active
**Model:** `EmployeeBenefit` ‚Äî tenant_id, employee_id, benefit_id, amount, effective_from, is_active
**Endpoints:** CRUD + assign to employees
**Frontend:** Benefit catalog + per-employee benefit assignment

### B5. Accounting Integration
**Model:** `AccountingConfig` ‚Äî tenant_id, integration_type (tally/quickbooks/manual), export_format, last_exported_at
**Endpoint:** Export payroll data as CSV/Excel in Tally-compatible format
**Frontend:** Accounting export screen with format selection

### B6. Statutory Reports
**Endpoints:**
- `GET /reports/statutory/pf` ‚Äî PF challan data
- `GET /reports/statutory/esi` ‚Äî ESI challan data
- `GET /reports/statutory/pt` ‚Äî Professional Tax report
- `GET /reports/statutory/form16` ‚Äî Form 16 data
**Frontend:** Statutory reports section in Reports screen

## Phase C: Communication (3 features)

### C1. Notification Engine
**Model:** `NotificationTemplate` ‚Äî tenant_id, name, event_type (leave_applied/leave_approved/attendance_missing/birthday/onboarding/welcome/payroll_generated), channel (email/sms/in_app), subject_template, body_template, is_active
**Model:** `NotificationPreference` ‚Äî tenant_id, user_id, event_type, email_enabled, sms_enabled, in_app_enabled
**Service:** Event-driven notification dispatch (trigger on API actions)
**Endpoints:** Template CRUD, preference update, send test
**Frontend:** Notification templates screen, preferences screen

### C2. Polls & Announcements
**Models:**
- `Announcement` ‚Äî tenant_id, title, body, priority (normal/important/urgent), publish_at, expires_at, is_active, created_by
- `Poll` ‚Äî tenant_id, question, options (JSONB), expires_at, is_anonymous, created_by
- `PollResponse` ‚Äî tenant_id, poll_id, employee_id, selected_option
**Endpoints:** CRUD + respond to poll
**Frontend:** Announcement board, Poll widget on dashboard

### C3. People Analytics
**Endpoints:**
- `GET /analytics/headcount` ‚Äî headcount trend over time
- `GET /analytics/attrition` ‚Äî attrition rate by department
- `GET /analytics/diversity` ‚Äî gender/department breakdown
- `GET /analytics/attendance-trends` ‚Äî attendance patterns
**Frontend:** Analytics dashboard with charts

## Phase D: Advanced (3 features)

### D1. Asset Tracking + Travel Desk
**Models:**
- `CompanyAsset` ‚Äî tenant_id, name, asset_code, category (laptop/phone/vehicle/other), serial_number, purchase_date, warranty_expiry, assigned_to, status (available/assigned/maintenance/retired)
- `TravelRequest` ‚Äî tenant_id, employee_id, destination, purpose, from_date, to_date, estimated_cost, status (pending/approved/rejected/completed), approved_by
**Endpoints:** CRUD for both
**Frontend:** Asset list, Asset assignment, Travel request form + list

### D2. Geo-Fencing + Selfie Clock-in
**Model:** `GeoFence` ‚Äî tenant_id, name, latitude, longitude, radius_meters, is_active
**Model:** `AttendancePhoto` ‚Äî tenant_id, employee_id, attendance_id, photo_path, latitude, longitude, captured_at
**Endpoints:** Clock-in with photo + location, verify geo-fence
**Frontend:** Mobile clock-in screen with camera + GPS

### D3. SSO + Mobile OTP
**Model:** `SSOConfig` ‚Äî tenant_id, provider (google/microsoft/saml), client_id, client_secret, is_active
**Model:** `OTPLog` ‚Äî tenant_id, phone, otp_code, expires_at, verified_at
**Endpoints:** SSO callback, send OTP, verify OTP
**Frontend:** SSO login buttons, OTP input screen

## Execution Order
1. Phase A (Core HR) ‚Äî foundation for all employee lifecycle features
2. Phase B (Finance) ‚Äî completes payroll ecosystem
3. Phase C (Communication) ‚Äî engagement and notifications
4. Phase D (Advanced) ‚Äî differentiators

## Files Pattern (per feature)
- `backend/app/models/<feature>.py`
- `backend/app/schemas/<feature>.py`
- `backend/app/api/v1/endpoints/<feature>.py`
- `backend/app/api/v1/router.py` (add router)
- `backend/alembic/versions/<migration>.py`
- `frontend/lib/screens/<section>/<feature>_screen.dart`
- `frontend/lib/core/router.dart` (add route)
- `frontend/lib/screens/main_shell.dart` (sidebar if needed)

## Verification
After each phase:
1. `docker exec apex_backend alembic upgrade head`
2. `docker compose up -d --build backend`
3. `cd frontend && flutter build web`
4. Test new screens and API endpoints
